# Understanding ATProto Lexicons

## What Is a Lexicon?

A Lexicon is a JSON schema definition document for the AT Protocol. Each file describes one or more typed definitions — records, query endpoints, procedure endpoints, or reusable type definitions — identified by an NSID (Namespace Identifier) like `app.bsky.feed.getTimeline`.

Every lexicon file shares this top-level structure:

```json
{
  "lexicon": 1,
  "id": "app.bsky.feed.getTimeline",
  "defs": { ... }
}
```

- `"lexicon": 1` — always present, always `1`; indicates the schema version of the lexicon format itself
- `"id"` — NSID in reverse-domain notation, matching the file path in the repo (`lexicons/app/bsky/feed/getTimeline.json`)
- `"defs"` — map of named definitions; the key `"main"` is the primary definition of the file

## Primary Definition Types

### Record

Records define data stored in user repositories — the persistent "nouns" of the system. Each record type corresponds to an AT Protocol collection, named after the lexicon's NSID.

```json
{
  "lexicon": 1,
  "id": "app.bsky.feed.like",
  "defs": {
    "main": {
      "type": "record",
      "description": "Record declaring a 'like' of a piece of subject content.",
      "key": "tid",
      "record": {
        "type": "object",
        "required": ["subject", "createdAt"],
        "properties": {
          "subject": { "type": "ref", "ref": "com.atproto.repo.strongRef" },
          "createdAt": { "type": "string", "format": "datetime" },
          "via": { "type": "ref", "ref": "com.atproto.repo.strongRef" }
        }
      }
    }
  }
}
```

Key fields:

- `"key"` — how the record's rkey is assigned: `"tid"` (timestamp-based ID generated by the client), `"any"` (application-controlled), or `"literal:..."` (a fixed string, used for singleton records)
- `"record"` — the actual object schema, with `"required"` and `"properties"`

Records do not include their own DID or collection in the stored data — those come from the repository path where they are stored (`at://did/app.bsky.feed.like/rkey`).

Record files across the two namespaces:

| File                      | Collection                    | Key   | Purpose                             |
| ------------------------- | ----------------------------- | ----- | ----------------------------------- |
| `feed/post.json`          | `app.bsky.feed.post`          | `tid` | Microblog post                      |
| `feed/like.json`          | `app.bsky.feed.like`          | `tid` | Like on any AT-URI                  |
| `feed/repost.json`        | `app.bsky.feed.repost`        | `tid` | Repost of another post              |
| `feed/postgate.json`      | `app.bsky.feed.postgate`      | `tid` | Embedding/quote controls for a post |
| `feed/threadgate.json`    | `app.bsky.feed.threadgate`    | `tid` | Reply gating rules for a thread     |
| `feed/generator.json`     | `app.bsky.feed.generator`     | `any` | Feed generator metadata             |
| `graph/follow.json`       | `app.bsky.graph.follow`       | `tid` | Follow relationship                 |
| `graph/block.json`        | `app.bsky.graph.block`        | `tid` | Block relationship                  |
| `graph/list.json`         | `app.bsky.graph.list`         | `tid` | List definition                     |
| `graph/listitem.json`     | `app.bsky.graph.listitem`     | `tid` | Membership in a list                |
| `graph/listblock.json`    | `app.bsky.graph.listblock`    | `tid` | Block all members of a list         |
| `graph/starterpack.json`  | `app.bsky.graph.starterpack`  | `tid` | Starter pack definition             |
| `graph/verification.json` | `app.bsky.graph.verification` | `tid` | Account verification relationship   |

### Query

Queries are read-only HTTP GET endpoints. Parameters map to URL query string arguments. They may be cached by the server.

```json
{
  "lexicon": 1,
  "id": "app.bsky.feed.getAuthorFeed",
  "defs": {
    "main": {
      "type": "query",
      "description": "Get a view of an actor's 'author feed' (post and reposts by the author). Does not require auth.",
      "parameters": {
        "type": "params",
        "required": ["actor"],
        "properties": {
          "actor": { "type": "string", "format": "at-identifier" },
          "limit": {
            "type": "integer",
            "minimum": 1,
            "maximum": 100,
            "default": 50
          },
          "cursor": { "type": "string" },
          "filter": {
            "type": "string",
            "description": "Combinations of post/repost types to include in response.",
            "knownValues": [
              "posts_with_replies",
              "posts_no_replies",
              "posts_with_media",
              "posts_and_author_threads",
              "posts_with_video"
            ],
            "default": "posts_with_replies"
          },
          "includePins": { "type": "boolean", "default": false }
        }
      },
      "output": {
        "encoding": "application/json",
        "schema": {
          "type": "object",
          "required": ["feed"],
          "properties": {
            "cursor": { "type": "string" },
            "feed": {
              "type": "array",
              "items": {
                "type": "ref",
                "ref": "app.bsky.feed.defs#feedViewPost"
              }
            }
          }
        }
      },
      "errors": [{ "name": "BlockedActor" }, { "name": "BlockedByActor" }]
    }
  }
}
```

The `"parameters"` block always has `"type": "params"` — a fixed marker, not a schema type. Parameters become `?actor=...&limit=50` in the URL. Array parameters repeat the key: `?others=did1&others=did2`.

The `"output"` block always has `"encoding": "application/json"` and a `"schema"` describing the response body.

### Procedure

Procedures are state-mutating HTTP POST endpoints. Input is a JSON body. They cannot be cached.

A simple side-effect-only procedure (no output):

```json
{
  "lexicon": 1,
  "id": "app.bsky.graph.muteActor",
  "defs": {
    "main": {
      "type": "procedure",
      "description": "Creates a mute relationship for the specified account. Mutes are private in Bluesky. Requires auth.",
      "input": {
        "encoding": "application/json",
        "schema": {
          "type": "object",
          "required": ["actor"],
          "properties": {
            "actor": { "type": "string", "format": "at-identifier" }
          }
        }
      }
    }
  }
}
```

A procedure with both input and output:

```json
{
  "lexicon": 1,
  "id": "app.bsky.feed.sendInteractions",
  "defs": {
    "main": {
      "type": "procedure",
      "description": "Send information about interactions with feed items back to the feed generator that served them.",
      "input": {
        "encoding": "application/json",
        "schema": {
          "type": "object",
          "required": ["interactions"],
          "properties": {
            "feed": { "type": "string", "format": "at-uri" },
            "interactions": {
              "type": "array",
              "items": {
                "type": "ref",
                "ref": "app.bsky.feed.defs#interaction"
              }
            }
          }
        }
      },
      "output": {
        "encoding": "application/json",
        "schema": { "type": "object", "properties": {} }
      }
    }
  }
}
```

Procedures without an `"output"` block return an empty 200 body. Procedures with an empty `"output"` schema (`"properties": {}`) declare that an output exists but is intentionally empty — a subtle distinction that signals this may grow in the future.

## Type System

### Primitive Types

| Type         | Notable Constraints                                                                               |
| ------------ | ------------------------------------------------------------------------------------------------- |
| `"boolean"`  | —                                                                                                 |
| `"integer"`  | `minimum`, `maximum`, `default`                                                                   |
| `"string"`   | `format`, `maxLength` (bytes), `maxGraphemes` (visual chars), `minLength`, `knownValues`, `const` |
| `"bytes"`    | Raw binary data                                                                                   |
| `"cid-link"` | Inline CID reference                                                                              |
| `"blob"`     | Binary upload; `accept` (MIME list), `maxSize` (bytes)                                            |
| `"unknown"`  | Escape hatch for arbitrary JSON — used where the record body is passed through opaquely           |

### String Formats

The `"format"` field gives a string semantic type beyond its syntax:

| Format            | Meaning                                            |
| ----------------- | -------------------------------------------------- |
| `"did"`           | Decentralized identifier (e.g. `did:plc:abc123`)   |
| `"handle"`        | User handle (e.g. `alice.bsky.social`)             |
| `"at-identifier"` | Either a DID or a handle                           |
| `"at-uri"`        | Full AT Protocol URI (`at://did/collection/rkey`)  |
| `"cid"`           | Content identifier (hash)                          |
| `"datetime"`      | ISO 8601 datetime                                  |
| `"language"`      | BCP-47 language tag                                |
| `"uri"`           | Generic URI (used for avatar URLs, external links) |
| `"nsid"`          | Namespace identifier                               |
| `"tid"`           | Timestamp-based record key                         |

### String Length Constraints: `maxLength` vs `maxGraphemes`

Text fields consistently apply two different length limits together. From `app.bsky.feed.post`:

```json
"text": {
  "type": "string",
  "maxLength": 3000,
  "maxGraphemes": 300,
  "description": "The primary post content. May be an empty string, if there are embeds."
}
```

- `maxLength` — byte limit (for storage/indexing safety)
- `maxGraphemes` — visual character limit (for display/UX)

This separation matters because multi-byte unicode (emoji, CJK characters) can differ dramatically between byte count and grapheme count. Descriptions follow the same pattern: `"maxGraphemes": 300, "maxLength": 3000`.

### `blob` Type

Used for image uploads. The blob is uploaded separately and referenced by its metadata:

```json
"avatar": {
  "type": "blob",
  "accept": ["image/png", "image/jpeg"],
  "maxSize": 1000000
}
```

### Reference Types

**`ref`** — points to a named definition by NSID fragment:

```json
{ "type": "ref", "ref": "app.bsky.actor.defs#profileViewBasic" }
```

Three ref forms:

- `"#typeName"` — local: another def in the same file
- `"com.atproto.repo.strongRef"` — refers to `main` of another file
- `"app.bsky.actor.defs#profileView"` — refers to a named def in another file

**`union`** — open polymorphic type; the value is exactly one of several refs. At runtime the chosen variant is identified by a `$type` field in the data:

```json
"embed": {
  "type": "union",
  "refs": [
    "app.bsky.embed.images",
    "app.bsky.embed.video",
    "app.bsky.embed.external",
    "app.bsky.embed.record",
    "app.bsky.embed.recordWithMedia"
  ]
}
```

Unions also appear in output schemas to express that a result may be one of several states:

```json
"thread": {
  "type": "union",
  "refs": [
    "app.bsky.feed.defs#threadViewPost",
    "app.bsky.feed.defs#notFoundPost",
    "app.bsky.feed.defs#blockedPost"
  ]
}
```

**`token`** — a named string constant, like an enum member. Tokens are defined in `defs` and referenced by their full NSID:

```json
"requestLess": {
  "type": "token",
  "description": "Request that less content like the given feed item be shown in the feed"
},
"requestMore": {
  "type": "token",
  "description": "Request that more content like the given feed item be shown in the feed"
}
```

A field references them via `knownValues`:

```json
"event": {
  "type": "string",
  "knownValues": [
    "app.bsky.feed.defs#requestLess",
    "app.bsky.feed.defs#requestMore",
    "app.bsky.feed.defs#interactionLike",
    "app.bsky.feed.defs#interactionRepost",
    "app.bsky.feed.defs#interactionReply",
    "app.bsky.feed.defs#interactionShare"
  ]
}
```

**`knownValues`** is an open enum: it lists known valid values but does not reject unknown ones. This makes it forward-compatible — servers and clients can add new values without breaking old parsers.

**`const`** — a boolean or string literal, used to create discriminant fields in union members:

```json
"notFoundPost": {
  "type": "object",
  "required": ["uri", "notFound"],
  "properties": {
    "uri": { "type": "string", "format": "at-uri" },
    "notFound": { "type": "boolean", "const": true }
  }
}
```

**`unknown`** — arbitrary JSON. Used when the record body is passed through without being parsed (e.g., `postView.record` holds the raw post record whose shape is known from a different lexicon):

```json
"record": { "type": "unknown" }
```

## Structural Conventions

### `defs.json` Files

Each namespace has a `defs.json` containing shared view types used across queries in that namespace. These files have no `"main"` — only named defs. The `app.bsky.feed.defs` file defines:

- `postView` — full post data for rendering in a feed or thread
- `feedViewPost` — a post within a feed, adding reply context and repost reason
- `threadViewPost` — a post within a thread, adding parent and replies
- `generatorView` — feed generator metadata for display
- `viewerState` — the authenticated user's relationship to a post (liked, reposted, etc.)
- `threadgateView` — threadgate rules, embedded in `postView`
- `skeletonFeedPost` — lightweight feed item used in feed skeleton protocol
- `interaction` — an interaction event to send back to a feed generator
- Token definitions: `requestLess`, `requestMore`, `interactionSeen`, `interactionLike`, etc.

The `app.bsky.graph.defs` file defines:

- `listViewBasic`, `listView` — minimal and full list representations
- `starterPackView`, `starterPackViewBasic`
- `listPurpose` — string type with `knownValues` pointing to purpose tokens
- `modlist`, `curatelist`, `referencelist` — purpose token definitions
- `relationship` — bi-directional graph relationship between two actors
- `notFoundActor` — error variant for unresolved actors
- `listViewerState` — authenticated user's relationship to a list

### `...View` vs `...ViewBasic` Pattern

A recurring convention: `Basic` variants omit expensive or rarely-needed fields. Compare `listViewBasic` vs `listView` from `graph/defs.json`:

```json
"listViewBasic": {
  "type": "object",
  "required": ["uri", "cid", "name", "purpose"],
  "properties": {
    "uri": { "type": "string", "format": "at-uri" },
    "cid": { "type": "string", "format": "cid" },
    "name": { "type": "string", "maxLength": 64, "minLength": 1 },
    "purpose": { "type": "ref", "ref": "#listPurpose" },
    "avatar": { "type": "string", "format": "uri" },
    "listItemCount": { "type": "integer", "minimum": 0 },
    "labels": { "type": "array", "items": { "type": "ref", "ref": "com.atproto.label.defs#label" } },
    "viewer": { "type": "ref", "ref": "#listViewerState" },
    "indexedAt": { "type": "string", "format": "datetime" }
  }
}
```

vs.

```json
"listView": {
  "type": "object",
  "required": ["uri", "cid", "creator", "name", "purpose", "indexedAt"],
  "properties": {
    "uri": { "type": "string", "format": "at-uri" },
    "cid": { "type": "string", "format": "cid" },
    "creator": { "type": "ref", "ref": "app.bsky.actor.defs#profileView" },
    "name": { "type": "string", "maxLength": 64, "minLength": 1 },
    "purpose": { "type": "ref", "ref": "#listPurpose" },
    "description": { "type": "string", "maxGraphemes": 300, "maxLength": 3000 },
    "descriptionFacets": { "type": "array", "items": { "type": "ref", "ref": "app.bsky.richtext.facet" } },
    "avatar": { "type": "string", "format": "uri" },
    "listItemCount": { "type": "integer", "minimum": 0 },
    "labels": { "type": "array", "items": { "type": "ref", "ref": "com.atproto.label.defs#label" } },
    "viewer": { "type": "ref", "ref": "#listViewerState" },
    "indexedAt": { "type": "string", "format": "datetime" }
  }
}
```

`listView` adds `creator` (a full `profileView`, expensive), `description`, and `descriptionFacets`. `listViewBasic` is used when embedding a list reference inside another view; `listView` is returned when the list itself is the subject of a query.

### ViewerState Pattern

Most entities include an optional `"viewer"` object describing the authenticated user's relationship to the content. From `feed/defs.json`:

```json
"postView": {
  "type": "object",
  "required": ["uri", "cid", "author", "record", "indexedAt"],
  "properties": {
    "uri": { "type": "string", "format": "at-uri" },
    "cid": { "type": "string", "format": "cid" },
    "author": { "type": "ref", "ref": "app.bsky.actor.defs#profileViewBasic" },
    "record": { "type": "unknown" },
    "replyCount": { "type": "integer" },
    "repostCount": { "type": "integer" },
    "likeCount": { "type": "integer" },
    "quoteCount": { "type": "integer" },
    "indexedAt": { "type": "string", "format": "datetime" },
    "viewer": { "type": "ref", "ref": "#viewerState" },
    "labels": { "type": "array", "items": { "type": "ref", "ref": "com.atproto.label.defs#label" } },
    "threadgate": { "type": "ref", "ref": "#threadgateView" }
  }
}
```

with the corresponding viewerState:

```json
"viewerState": {
  "type": "object",
  "description": "Metadata about the requesting account's relationship with the subject content. Only has meaningful content for authed requests.",
  "properties": {
    "repost": { "type": "string", "format": "at-uri" },
    "like": { "type": "string", "format": "at-uri" },
    "bookmarked": { "type": "boolean" },
    "threadMuted": { "type": "boolean" },
    "replyDisabled": { "type": "boolean" },
    "embeddingDisabled": { "type": "boolean" },
    "pinned": { "type": "boolean" }
  }
}
```

When the viewer has liked a post, `viewer.like` contains the AT-URI of their own like record. This lets the client render the like button as active and allows undoing the like by deleting that specific record.

### Pagination Convention

All list-returning queries use cursor-based pagination with a consistent shape. `getTimeline` is the minimal example:

```json
"main": {
  "type": "query",
  "parameters": {
    "type": "params",
    "properties": {
      "algorithm": { "type": "string" },
      "limit": { "type": "integer", "minimum": 1, "maximum": 100, "default": 50 },
      "cursor": { "type": "string" }
    }
  },
  "output": {
    "encoding": "application/json",
    "schema": {
      "type": "object",
      "required": ["feed"],
      "properties": {
        "cursor": { "type": "string" },
        "feed": {
          "type": "array",
          "items": { "type": "ref", "ref": "app.bsky.feed.defs#feedViewPost" }
        }
      }
    }
  }
}
```

- Client omits `cursor` on first request
- Server includes `cursor` in response if more pages exist
- Client passes the returned `cursor` on the next request
- When the server returns no `cursor`, there are no more pages

`cursor` is always optional in both input and output. `limit` always has `minimum`, `maximum`, and `default`.

### Strong References

References to specific record versions use `com.atproto.repo.strongRef`, which bundles an AT-URI with a CID. This pins the reference to a specific content hash, enabling versioned, tamper-evident linking.

From `app.bsky.feed.like`:

```json
"subject": { "type": "ref", "ref": "com.atproto.repo.strongRef" }
```

Whereas a `follow` record references only a DID (not a versioned record):

```json
"subject": { "type": "string", "format": "did" }
```

The choice between `strongRef` and a plain `did`/`at-uri` is deliberate: likes and reposts need to pin the exact version of the content they reference; follows point to an identity that can evolve.

### Relationship Records Reference the Subject Directly

Graph relationship records (`follow`, `block`) store their subject as the only meaningful payload — they have no content beyond who is being followed/blocked:

```json
{
  "lexicon": 1,
  "id": "app.bsky.graph.follow",
  "defs": {
    "main": {
      "type": "record",
      "description": "Record declaring a social 'follow' relationship of another account. Duplicate follows will be ignored by the AppView.",
      "key": "tid",
      "record": {
        "type": "object",
        "required": ["subject", "createdAt"],
        "properties": {
          "subject": { "type": "string", "format": "did" },
          "createdAt": { "type": "string", "format": "datetime" },
          "via": { "type": "ref", "ref": "com.atproto.repo.strongRef" }
        }
      }
    }
  }
}
```

The fact that _user A follows user B_ is encoded by _the existence of a follow record in A's repo_ where `subject` is B's DID. Deleting that record ends the follow. There is no separate "unfollow" procedure lexicon.

### Error Declarations

Queries and procedures declare expected error names alongside the main definition:

```json
"errors": [
  { "name": "BlockedActor" },
  { "name": "BlockedByActor" }
]
```

With optional descriptions:

```json
"errors": [
  {
    "name": "ActorNotFound",
    "description": "the primary actor at-identifier could not be resolved"
  }
]
```

Error names are PascalCase. Descriptions are inconsistently present — some errors have them, many do not.

### Union-Based Error Variants in Output

For queries where a missing or blocked resource is a normal, non-error case, the output schema uses a union to represent possible states rather than an HTTP error code. From `getPostThread`:

```json
"output": {
  "encoding": "application/json",
  "schema": {
    "type": "object",
    "required": ["thread"],
    "properties": {
      "thread": {
        "type": "union",
        "refs": [
          "app.bsky.feed.defs#threadViewPost",
          "app.bsky.feed.defs#notFoundPost",
          "app.bsky.feed.defs#blockedPost"
        ]
      }
    }
  }
}
```

The client inspects `$type` in the response to know which variant it received. `notFoundPost` and `blockedPost` are inline "error" states that still return HTTP 200.

### Threadgate Rules as a Union Array

The threadgate record shows how a union inside an array can model a rule system:

```json
"allow": {
  "description": "List of rules defining who can reply to this post. If value is an empty array, no one can reply. If value is undefined, anyone can reply.",
  "type": "array",
  "maxLength": 5,
  "items": {
    "type": "union",
    "refs": ["#mentionRule", "#followerRule", "#followingRule", "#listRule"]
  }
}
```

Each rule variant is a small object, some with no properties (acting as marker types):

```json
"mentionRule": { "type": "object", "description": "Allow replies from actors mentioned in your post.", "properties": {} },
"followerRule": { "type": "object", "description": "Allow replies from actors who follow you.", "properties": {} },
"listRule": {
  "type": "object",
  "description": "Allow replies from actors on a list.",
  "required": ["list"],
  "properties": { "list": { "type": "string", "format": "at-uri" } }
}
```

### Deprecation

Deprecated fields carry a `"description"` beginning with `"DEPRECATED"`. The field remains in the schema for backward compatibility but clients should not produce it:

```json
"entities": {
  "type": "array",
  "description": "DEPRECATED: replaced by app.bsky.richtext.facet.",
  "items": { "type": "ref", "ref": "#entity" }
}
```

The `entity` and `textSlice` types still exist in `post.json` as local defs, referenced only by the deprecated field, so old data can still be parsed.

## Cross-Namespace References

The feed and graph lexicons heavily reference types from other namespaces rather than duplicating definitions. All cross-file references use the absolute `"namespace.path#defName"` form.

| Used in   | References                             | Purpose                                       |
| --------- | -------------------------------------- | --------------------------------------------- |
| `feed/*`  | `app.bsky.actor.defs#profileView`      | Author field on most view types               |
| `feed/*`  | `app.bsky.actor.defs#profileViewBasic` | Lighter author field (e.g., in `postView`)    |
| `feed/*`  | `app.bsky.graph.defs#listViewBasic`    | List reference in `threadgateView`            |
| `feed/*`  | `app.bsky.richtext.facet`              | Text annotation for mentions, links, hashtags |
| `feed/*`  | `app.bsky.embed.*`                     | Image, video, external, record embed types    |
| `graph/*` | `app.bsky.actor.defs#profileView`      | Creator field on list/starterpack views       |
| `graph/*` | `app.bsky.feed.defs#generatorView`     | Feeds embedded inside a starter pack          |
| Both      | `com.atproto.label.defs#label`         | Moderation labels on any content              |
| Both      | `com.atproto.repo.strongRef`           | Versioned reference to a specific record      |
| Both      | `com.atproto.label.defs#selfLabels`    | Self-applied content warnings                 |

`app.bsky.actor.defs` sits at the center — nearly every view type embeds a profile view for the content author. The reference graph goes: `feed/graph → actor → (back to feed via starterpack)`, with `com.atproto.*` types at the base providing protocol-level primitives.

## XRPC Transport

Lexicons describe schema; XRPC is the HTTP transport:

- Queries → `GET /xrpc/{nsid}` with parameters as query string
- Procedures → `POST /xrpc/{nsid}` with JSON body
- All response bodies are `application/json`
- Errors return `{ "error": "PascalCaseName", "message": "..." }`
- Authentication: OAuth (current standard) or legacy JWT tokens; inter-service calls use short-lived JWTs with an `lxm` claim that scopes the token to a specific NSID
- Pagination is always cursor-based; there is no offset pagination in the protocol
- Array parameters in query strings repeat the key: `?others=did1&others=did2`

## Are These Files Hand-Written or Generated?

The lexicon JSON files are **hand-written** — they are the source of truth from which client SDKs and server stubs are generated (via `@atproto/lex-cli` / `@atproto/lex`). Evidence:

1. **Inconsistent description coverage** — some fields have detailed descriptions, many have none, and the prose style varies across files. A generator produces uniform coverage or none at all.

2. **Non-canonical field ordering** — properties are not alphabetically sorted; they reflect human authoring choices (required fields first, optional fields grouped by concern). `uri`, `cid`, `author` appear together because they belong together conceptually.

3. **Deprecation in prose** — `"DEPRECATED: replaced by app.bsky.richtext.facet."` is an editorial note inside a description string, not a structured field. Generated schemas would use a dedicated `deprecated: true` flag.

4. **No generator metadata** — generated JSON files typically carry a header comment or metadata field naming the tool and input source. None is present.

5. **`defs.json` groupings reflect design judgment** — what becomes a standalone file vs. a def inside `defs.json` (e.g., why `postgate` is a record file but `viewerState` is a def) reflects authorial decisions about identity and reuse, not mechanical decomposition.

6. **Selective application of `knownValues`** — some semantically enum-like fields use `knownValues`; others with similar character do not. Human authors make case-by-case decisions; generators apply rules consistently.

7. **Typos in descriptions** — `graph/list.json` has `"curration-oriented"` (two r's) in its description, a human error that a generator would not introduce.
